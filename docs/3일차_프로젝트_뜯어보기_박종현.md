# 과제: 3일차_프로젝트_뜯어보기.md

## 1. 각 디렉터리의 역할 및 작성해야 할 파일의 내용

* app/core/
  * 역할: 애플리케이션 전체에서 공통으로 사용되는 핵심 설정, 보안 인증, 상수 환경 변수 등을 관리하는 공간입니다.
  * 작성 내용: 환경 변수 로드 세팅(config.py), JWT 토큰 인증 스크립트, 보안 비밀키 설정 파일 등이 작성됩니다.
* app/models/
  * 역할: 데이터베이스의 테이블 구조를 파이썬 클래스로 정의하고 관리하는 ORM 모델 공간입니다.
  * 작성 내용: SQLAlchemy의 Base 객체를 상속받아 실제 DB 테이블 이름, 기본키(PK), 외래키(FK), 데이터 타입 및 관계(relationship)를 선언하는 모델 코드가 작성됩니다.
* app/schemas/
  * 역할: 클라이언트와 서버 간에 데이터를 주고받을 때 입력값과 출력값의 규격을 제한하는 데이터 검증(DTO) 공간입니다.
  * 작성 내용: Pydantic의 BaseModel을 상속받아 API 요청 데이터(Request Body) 및 응답 데이터(Response Body)의 필드명과 데이터 타입을 정의하는 코드가 작성됩니다.
* app/repositories/
  * 역할: 데이터베이스에 직접 접근하여 CRUD(생성, 조회, 수정, 삭제) 쿼리 연산만을 전담하는 데이터 접근 계층입니다.
  * 작성 내용: SQLAlchemy 세션을 활용하여 실제 DB 테이블에 데이터를 insert, select, update, delete 하는 순수 DB 처리 함수들이 작성됩니다.
* app/services/
  * 역할: 애플리케이션의 핵심 비즈니스 로직과 알고리즘 가공 처리를 담당하는 서비스 계층입니다.
  * 작성 내용: 컨트롤러(API)와 데이터베이스(Repository) 사이에서 데이터를 가공하고, 비즈니스 규칙을 검증하며, 외부 API 연동이나 예외 처리를 총괄하는 핵심 로직 코드가 작성됩니다.
* app/apis/
  * 역할: 클라이언트의 HTTP 요청을 가장 먼저 받아 알맞은 서비스로 넘겨주는 엔드포인트(라우터) 계층입니다.
  * 작성 내용: APIRouter 인스턴스를 생성하고 @router.get, @router.post 등의 데코레이터를 사용하여 URL 경로를 지정하고 요청을 처리하는 핸들러 함수가 작성됩니다.

## 2. 각 파일의 역할

* app/main.py
  * 역할: FastAPI 애플리케이션이 실행되는 시작점이자 진입점입니다.
  * 내용: FastAPI 인스턴스를 생성하고, 미들웨어(CORS 등)를 설정하며, 각 도메인별로 분리된 APIRouter들을 하나로 묶어 통합 등록하는 서버 인프라 코드가 작성됩니다.
* app/core/config.py
  * 역할: 애플리케이션의 환경 설정 정보를 한곳에서 중앙 제어하는 파일입니다.
  * 내용: Pydantic의 Settings 인터페이스 등을 활용하여 .env 파일에 정의된 데이터베이스 접속 주소, 비밀키, 가동 환경(개발/운영) 등의 시스템 환경 변수를 로드하는 코드가 작성됩니다.
* pyproject.toml
  * 역할: 파이썬 프로젝트의 메타데이터와 패키지 의존성 및 빌드 시스템을 정의하는 통합 설정 파일입니다.
  * 내용: 프로젝트 이름, 버전, 파이썬 버전 요구사항과 uv 또는 poetry 같은 최신 패키지 매니저가 사용하는 서드파티 라이브러리 목록 및 도구 설정들이 기록됩니다.
* uv.lock
  * 역할: 프로젝트에 설치된 정확한 패키지 버전을 고정하여 어디서나 동일한 환경을 보장하는 잠금 파일입니다.
  * 내용: pyproject.toml에 명시된 라이브러리들과 그에 의존하는 하위 패키지들의 정확한 버전, 체크섬 정보가 자동으로 기록되어 팀원 간 개발 환경 동기화를 보장합니다.

## 3. 데이터베이스 연결 설정

* app/core/db/databases.py 관련 내용:
  * 애플리케이션이 데이터베이스에 안전하게 접속하고 통신할 수 있도록 다리 역할을 하는 연결 세팅 파일입니다.
  * 1. create_engine: 설정된 DB 주소(URL)를 바탕으로 데이터베이스와의 물리적인 연결 커넥션을 생성합니다.
  * 2. sessionmaker: 개발자가 데이터베이스 변경 연산(트랜잭션)을 제어할 수 있는 창구인 데이터베이스 세션(Session)을 생성하는 인스턴스를 빌드합니다.
  * 3. get_db(): FastAPI의 의존성 주입(Depends) 시스템용 함수로, API 요청이 오면 세션을 열어주고(yield), 연산 처리가 끝나면 에러 발생 여부와 상관없이 finally 구문을 통해 DB 커넥션을 안전하게 닫아주는(db.close()) 자원 관리 로직이 구현됩니다.

## 4. SQLAlchemy를 활용하여 모델을 작성하고 alembic을 활용하여 마이그레이션을 수행하는 방법

1. SQLAlchemy 모델  app/models/ 디렉터리 하위에 엔티티별 py 파일을 생성한 뒤, Base 객체를 상속받는 클래스를 선언합니다. __tablename__으로 테이블명을 지정하고, Column, Integer, String 등을 이용해 ERD 구조에 맞는 스키마를 선언합니다.
2. Alembic 마이그레이션 수행 방법 
   * 터미널에 `alembic init alembic`을 입력하여 마이그레이션 초기 환경을 구성합니다.
   * `alembic/env.py` 파일 내 `target_metadata`에 작성한 SQLAlchemy Base.metadata를 연동합니다.
   * `alembic revision --autogenerate -m "메시지"` 명령어를 입력하여 변경된 모델 구조를 감지한 마이그레이션 스크립트 파일을 자동 생성합니다.
   * `alembic upgrade head` 명령어를 최종 실행하여 데이터베이스에 스키마를 동기화하고 업데이트합니다.

## 5. API 구현 흐름

1. [클라이언트 요청] 브라우저 또는 클라이언트가 특정 HTTP Method와 URL로 요청을 전송합니다.
2. [라우터 진입] app/main.py를 거쳐 app/apis/에 정의된 라우터 엔드포인트에 요청이 매핑됩니다.
3. [데이터 검증] app/schemas/의 Pydantic BaseModel을 통과하며 데이터 유효성 및 타입 검사가 자동 수행됩니다.
4. [비즈니스 로직] app/services/ 계층으로 데이터가 넘어가 필요한 핵심 연산 및 정책 규칙 처리가 수행됩니다.
5. [DB 접근 및 처리] app/repositories/ 계층에서 get_db()로 주입받은 세션을 통해 app/models/ 테이블 객체에 접근하여 CRUD 쿼리를 실행합니다.
6. [최종 응답] 처리 결과 데이터가 다시 셰마 규격에 맞춰 JSON 형태로 변환되어 클라이언트에게 반환됩니다.
